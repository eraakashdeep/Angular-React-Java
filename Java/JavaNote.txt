Q1. Why String is immutable in java ? (value Can't be changed after creation or created and string is not designed for modification.)

Q2. Difference between String and StringBuffer ?
Ans-> String : immutable, Stored in String Pool, Any Changes -> new object (value can't be changed once created)
      String Buffer : mutable, Changes happen in same object, Thread-safe(synchronized) (It can be modified)

      Ex:- 
      String s = "hello";
      s.concat(" world"); // creates new object

      StringBuffer sb = new StringBuffer("hello");
      sb.append(" world"); //modifies same object

Q3. Difference between fail-fast and fail-safe iterators ?
Ans-> Fail-fast : Fast but unsafe (throws exception), works on original collection (ex:- ArrayList, HashMap)
      Fail-safe : Safe but uses extra memory (no exception), works on copy collection (ex:- CopyOnWriteArrayList, ConcurrentHashMap)

Q7. Can we override private methods in java ?
Ans-> No, we cannot override a private method in Java. A private method is not visible 
to subclasses, so it cannot be inherited & if a method isn’t inherited, it cannot be overridden.

Q8. Can we override static method ? what is static method.
Ans-> No, Static methods cannot be overridden in Java. [they are hidden, not overridden.]

Note: 
- Static methods belong to the class, not the object.
- Overriding works on runtime polymorphism (objects).
- Static methods are resolved at compile time, so no overriding.
- Static Keyword used for variables, methods, blocks, nested classes.
ex: public class Parent {
    public static void show() {
        System.out.println("Parent");
    }
}

public class Child extends Parent {
    public static void show() {
        System.out.println("Child");
    }
}

Parent obj = new Child();
obj.show();  // Output: Parent ➡️ Even though object is Child, it calls Parent method ➡️ This is method hiding, not overriding.

Q10. OOPs concept in real time ?
Ans-> 
a. Encapsulation (Data Hiding) : it helps in data security and controlled access using getters/setters.
ex: In a banking system, account balance is private, You access it via methods like deposit() / withdraw() 
b. Abstraction (Hiding complexity) : it reduces complexity and improves maintainability by exposing only required features. 
ex: Using a car — you drive without knowing engine internals
c. Inheritance (Reusability) : One class inherits properties of another 
ex: Employee → Manager, Developer
class Employee { void work() {} }
class Developer extends Employee { void code() {} }
d. Polymorphism (Many forms) : Same method, different behavior
ex: Payment system → pay() behaves differently for UPI / Card / NetBanking

Note: For example, in a payment system, we use abstraction to define payment methods, 
      inheritance to extend different payment types, encapsulation to secure transaction data, 
      and polymorphism to process payments dynamically.

Q21. Name of inbuilt functional interfaces in Java8?
Ans-> Java 8 introduced functional interfaces mainly in 👉 java.util.function package
      a. Predicate<T> - Takes input, returns boolean - ex: Predicate<Integer> isEven = x -> x % 2 == 0;
      b. Function<T, R> - Takes input, returns output - ex: Function<String, Integer> length = str -> str.length(); 
      c. Consumer<T> - Takes input, returns nothing - ex: Consumer<String> print = str -> System.out.println(str);
      d. Supplier<T> - No input, returns value - ex: Supplier<Double> random = () -> Math.random();

Q26. How to fix memory leak problems ?
Ans-> A memory leak happens when: Objects are no longer needed but are still referenced, 
      so the Garbage Collector (GC) cannot remove them.
      Fix:
      - Remove unnecessary references
      - Use proper collection handling
      - Close resources properly (Use try-with-resources)
      - Clean up ThreadLocal & Avoid unbounded caches
      - Deregister listeners
      - Use Weak References
      Tools to detect memory leaks
      - MAT(Memory Analyzer Tool)
      - JProfiler
      - VisualVM
      Memory leaks Cause
      - outofMemoryError
